## 1. 문서 개요

### 1.1 목적
팀장급 이상 관리자가 **주요 일정·긴급 이슈·중요 결정사항**을 메모 형식으로 빠르게 공유하고, 실시간 동기화·검색·알림으로 업무 공백 없이 협업할 수 있는 모바일 앱(WISM)을 개발하기 위한 기획 및 개발 계획을 정의한다.

### 1.2 범위
- **포함:** 인증/로그인, 메모(일정·이슈·결정사항), 첨부파일, 캘린더, 알림/공지, 검색·필터·북마크, 읽음 확인, 프로필, 클라이언트-서버 REST 연동
- **제외(추후 논의):** 결재/전자문서, 근태, 메신저(실시간 채팅), 외부 SSO 연동
- **향후 확장(로드맵):** 고객/클라이언트 히스토리 조회, AI 챗봇·자동 요약 (→ "12. 향후 발전 방향")

### 1.3 용어
| 용어 | 설명 |
| --- | --- |
| WISM | Wintek Insight System Manager — 본 앱의 명칭 |
| 메모(Memo) | 업무 공유의 기본 단위. 중요도·카테고리·프로젝트·담당자 태그·예약일·댓글·읽음확인을 가짐 |
| 카테고리 | 메모 분류: 일정 / 이슈 / 결정사항 / 회의록 / 기타 |
| 중요도 | 긴급 / 일반 (카테고리와 별개 축) |
| 일정 | 카테고리가 "일정"이고 `scheduledDate`가 있는 메모. 캘린더에 표시 (별도 엔티티 아님) |
| 사업(프로젝트) | 메모에 연결되는 사업. **기존 사업 데이터(사업명↔주관업체)에서 선택**(자유 입력 X), 선택 시 주관업체 자동 연결. 2차 고객 히스토리의 토대 |
| 담당자 태그 | 메모에 연결된 `@담당자`. **userData 검색으로 실제 유저를 선택**(자유 입력 X), userId 참조. 멘션 알림과 연동 |
| 읽음 확인 | 메모를 확인한 관리자 수(readBy/totalReaders) 추적 |
| 관리자 | 본 앱의 사용자 = 팀장급 이상 |

## 2. 프로젝트 개요

### 2.1 배경 / 문제점 (초기 제안서 기준)
관리자들이 주요 일정·이슈를 **개별 메신저·이메일·전화로 산발적으로 공유**하면서 다음 문제가 발생:
- 긴급 이슈 발생 시 관련 관리자에게 **일일이 개별 연락** → 전달 지연·누락 위험
- 과거 결정사항·이슈 히스토리가 메신저/이메일에 흩어져 **추적·검색 곤란**
- 정보가 일부 관리자에게만 전달되어 **업무 공백** 발생
- 전사 업무 가시성 부족 → 일정 조율·의사결정 속도 저하, 동일 내용 중복 문의

### 2.2 해결책 / 목표
1. **단일 입력 → 전체 전파** — 메모 1건 작성으로 전체 관리자에게 실시간 동기화 (개별 연락 제거)
2. **빠른 기록** — 제목·카테고리·프로젝트·담당자 태그로 30초 이내 구조화 입력
3. **즉시 파악** — 검색·필터·북마크·읽음 확인으로 필요한 정보에 즉시 접근
4. **이력 관리** — 결정사항·이슈 자동 기록 → 참조·책임 추적·인수인계에 활용
5. **중앙 집중 관리** — 회사 홈페이지 DB 연동으로 정보 일원화

### 2.3 기대 효과
- **업무 신속성:** 개별 연락·회신 대기 제거, 실시간 공유로 빠른 진행
- **업무 효율성:** 전사 가시성·투명성 확보, 협업 강화, 결정사항 이력 관리

### 2.4 대상 사용자
- **팀장급 이상 관리자 전용** (전 직원 대상 아님)
- 로그인 식별자: **사번 기반, 기존 홈페이지 DB 계정 연동** (9-2 확정)
- 권한: **단일 등급** — 모든 관리자가 동일 권한. 추후 분리 대비 `role` 필드만 확보 (9-3 확정)

## 3. 핵심 기능 정의

### 3.1 인증 / 로그인
- 사번 + 비밀번호 기반 로그인 (REST `/auth/login`)
- 토큰 기반 인증 (JWT Access + Refresh), 자동 로그인/세션 유지
- 로그아웃, 토큰 만료 시 재로그인 유도

### 3.2 메모 (핵심)
- 작성/조회/수정/삭제 (CRUD) — 작성은 FAB → 중앙 모달, 30초 이내 입력 목표
- 입력 필드: **제목 · 중요도(긴급/일반) · 카테고리(일정/이슈/결정사항/회의록/기타) · 사업(프로젝트) · 내용 · 담당자 태그(@)**
- **사업(프로젝트) = 기존 사업 데이터에서 선택**(자동완성): 사업명 입력 → `사업 데이터` 검색 → 후보 드롭다운 → 선택 시 **주관업체 자동 연결**(자유 입력 아님, projectId 참조). 2차 고객 히스토리의 토대
- **담당자 태그 = 유저 검색 자동완성**(하이웍스 참조 방식): 이름 입력 → `userData`에서 검색 → `이름 · 부서 · 직급` 후보 드롭다운 → 선택 시 칩 추가 (자유 입력 아님, userId 참조). `flutter_typeahead`로 디바운스 비동기 검색. 동명이인은 부서·직급으로 구분
- 카테고리가 "일정"일 때만 **일정 날짜(scheduledDate)** 입력 노출 → 캘린더 연동
- **댓글** 작성 (상세화면 하단 입력) — 일반/상태(확인완료) 타입
- **읽음 확인**: 상세화면 "읽음 확인" 버튼 → readBy/totalReaders 카운트 증가
- **북마크**: 중요 메모 상단 고정·북마크 탭 모음
- **검색**: 제목·내용·프로젝트명 통합 검색 / **필터**: 카테고리 칩, 날짜·담당자 / **정렬**: 최신순 등
- **소유자 전용**: 본인 메모일 때만 상세화면에서 수정/삭제 노출
- 목록 카드: 카테고리·긴급·프로젝트 배지, 작성자·시간·조회수·댓글수, 북마크 토글
- **첨부파일**: 이미지(`image_picker`)·문서(`file_picker`) 선택 → `multipart/form-data` 업로드. 이미지 미리보기, 일반 파일 다운로드/열기
  - 저장: **홈페이지 웹서버 디스크 저장 + DB에 메타데이터**(원본명·타입·크기·경로) 기록
  - 다운로드 URL도 인증 토큰으로 보호 (관리자 전용)
  - 허용 형식: **이미지·PDF·한글(HWP)·Word·PPT·Excel** / **파일당 10MB** / **메모당 3개** / **이미지 자동 압축** (9-5 확정)

### 3.3 캘린더
- 대시보드에서 진입하는 **월간 그리드 모달** (탭 아님)
- `scheduledDate`가 있는 메모를 날짜별 도트(카테고리 색)로 표시
- 카테고리 칩 필터, 날짜 선택 시 해당일 일정 리스트 조회 → 메모 상세 이동
- 대시보드에는 "이번 주 일정" 주간 스트립도 별도 제공

### 3.4 알림 / 공지
- **푸시 알림**(FCM) 수신 → 탭하면 해당 메모로 딥링크(상세 오픈)
- **알림 패널**: 헤더 종 아이콘 → 슬라이드인, 누적·읽음 처리, 미확인 배지 카운트
- 알림 유형: **긴급(urgent) · 댓글(comment) · 멘션(mention) · 확인(status)**
- 멘션 알림: 메모 담당자로 지정된 유저에게 푸시 발송 (9-9 확정)
- 공지: 단일 등급이므로 별도 발송 권한 구분 없음 (추후 상위 권한 분리 시 재검토)

### 3.5 프로필 / 설정
- **프로필 카드**: 이름·부서·직책·이메일·전화번호·사진 조회, [편집] 진입
- **활동 통계**: 내 메모 / 댓글 / 북마크 개수
- **설정(프로필 화면 내 인라인)**: 푸시 알림 on/off 토글, 버전 정보(v1.0.0)
- **로그아웃**
- **프로필 편집(바텀시트)**: 이름·직급·부서·이메일·전화번호 수정 (사번 편집 없음). **사진은 회사 내부 서버 연동(읽기 전용) — 앱에서 변경 불가**
- ※ 설정은 별도 화면이 아니라 프로필 화면 하단에 포함됨. 항목 확장 시 별도 설정 화면 분리 검토

## 4. 정보 구조 / 화면 목록

> **디자이너 산출물(`a.Design/02Main`) 기준으로 정렬됨.** 화면은 `P-NN` 코드 체계로 관리한다.

### 4.1 내비게이션 구조
- 앱은 **단일 메인 셸(P-02)** 안에 **하단 탭 5개**를 두고, 작성/상세/알림/캘린더/프로필편집은 **오버레이(모달·풀스크린·슬라이드인)**로 띄운다.
- **캘린더는 탭이 아니라** 대시보드에서 여는 **풀스크린 모달**이다.
- 메모 목록은 **범위(전체/내 메모/북마크)별로 분리**되며, 카테고리는 목록 내부의 **필터 칩**이다.

## 6. 아키텍처

### 6.1 레이어 구조 (Clean Architecture + MVVM)
```
Presentation (UI)
  └─ Screen (Widget) ── ViewModel (Riverpod Notifier)
            │
Domain
  └─ Entity / UseCase / Repository(interface)
            │
Data
  └─ Repository(impl) ── DataSource(Remote: Dio/Retrofit, Local: storage)
```
- UI는 ViewModel의 상태만 구독 → 비즈니스 로직과 화면 분리
- Repository 인터페이스로 데이터 출처(원격/로컬) 추상화 → **사내 서버 이전 시 DataSource만 교체**

### 6.2 폴더 구조(안)
```
lib/
├─ main.dart
├─ app/                # MaterialApp, 라우터, 테마, DI 설정
├─ core/               # 상수, 에러, 네트워크(Dio/인터셉터), 유틸
├─ features/
│  ├─ auth/            # 로그인/토큰
│  │  ├─ data/ domain/ presentation/
│  ├─ memo/            # 메모/게시글
│  ├─ calendar/        # 캘린더
│  ├─ notification/    # 알림/공지
│  └─ profile/         # 프로필
└─ shared/             # 공통 위젯, 테마, 디자인 토큰
```

### 6.3 백엔드 연동 전략
- **최종 인프라:** 기존 회사 홈페이지 **웹서버·DB를 그대로 활용**하고, DB에 WISM 전용 테이블을 추가 (추가 인프라 비용 X)
- **개발 단계:** AWS 임시 서버로 병행 개발 → 4개월차에 사내(홈페이지) 서버·DB로 연동 전환
- 모든 통신은 **REST/JSON over HTTPS**. `BASE_URL`을 환경(.env / flavor)으로 분리 → **AWS 임시 ↔ 사내 서버 전환을 빌드 설정만으로** 처리
- 공통 인터셉터: Authorization 헤더 자동 첨부, 401 시 Refresh 토큰으로 재시도
- 에러 표준화: `{ code, message }` 형태 응답 규약 (서버와 협의 필요)
- ⚠️ 기존 홈페이지 DB 스키마·인증 체계와의 정합성, 외부망 접근 정책은 **서버팀 협의 선행 항목**

## 7. 데이터 모델 (초안)

서버 API 스펙 확정 전 클라이언트 관점 초안. **디자인의 메모 구조에 정렬** (일정은 별도 엔티티가 아니라 메모의 한 종류).

```
User    { id, employeeNo, name, email, phone, position, dept, photoUrl, role }
Project { id, name, clientName }   // 기존 사업 데이터(사업명↔주관업체), MVP는 읽기 전용 조회
Memo    { id, title, content,
          priority,          // 긴급 | 일반
          category,          // 일정 | 이슈 | 결정사항 | 회의록 | 기타
          project,           // {projectId, name, clientName} — 사업 데이터에서 선택(주관업체 자동연결)
          assignees[],       // 담당자 태그: [{userId, name}] — userData에서 선택, mention 알림 연동
          scheduledDate?,    // category=일정 일 때 캘린더 표시
          authorId, date, time, createdAt, updatedAt,
          readBy, totalReaders,   // 읽음 확인 카운트
          bookmarked, commentCount, attachments[] }
Attachment { id, memoId, fileName, mimeType, size, url, uploadedAt }
Comment { id, memoId, authorId, content, type, createdAt }  // type: comment | status
ReadReceipt  { memoId, userId, readAt }        // 누가 읽었는지(읽음 확인)
Bookmark     { memoId, userId }                // 사용자별 북마크
Notification { id, type, title, content, memoId, read, createdAt }
                                               // type: urgent | comment | mention | status
Device       { userId, fcmToken, platform }    // 푸시 대상
```

> 카테고리는 고정값이면 상수, 가변이면 `Category{id,name,order}` 테이블로.

### 주요 API 엔드포인트(안)
| 기능 | 메서드 | 경로 |
| --- | --- | --- |
| 로그인 | POST | /auth/login |
| 토큰 갱신 | POST | /auth/refresh |
| 유저 검색(담당자 태그) | GET | /users?q={이름} → [{id,name,dept,position,photoUrl}] |
| 사업 검색(프로젝트) | GET | /projects?q={사업명} → [{id,name,clientName}] |
| 메모 목록 | GET | /memos?category=&priority=&q=&scope=&sort=&page= |
| 메모 상세 | GET | /memos/{id} |
| 메모 작성/수정/삭제 | POST/PUT/DELETE | /memos , /memos/{id} |
| 읽음 확인 | POST | /memos/{id}/read |
| 북마크 토글 | PUT/DELETE | /memos/{id}/bookmark |
| 댓글 | GET/POST/PUT/DELETE | /memos/{id}/comments |
| 첨부 업로드 | POST (multipart) | /memos/{id}/attachments |
| 첨부 삭제 | DELETE | /attachments/{id} |
| 첨부 다운로드 | GET (인증) | /attachments/{id}/download |
| 캘린더(기간 일정) | GET | /memos?from=&to=&category=일정 |
| 알림 목록 | GET | /notifications |
| 알림 읽음 | PUT | /notifications/{id}/read |
| FCM 토큰 등록 | POST | /devices |

> `scope` = all / my / bookmarks (전체·내메모·북마크 탭 대응)

## 12. 향후 발전 방향 (초기 제안서 로드맵)

1차 출시 이후 단계적 확장 — **현재 범위(MVP)에서는 제외**하되 구조 설계 시 고려:

- **고객/클라이언트 히스토리 조회** (제안서 시나리오 2, **2차 확정**): 업체별 고객 데이터·과거 사업 진행현황·메모 히스토리 조회 → 영업/응대 관리.
  - 신규 엔티티: `Client(업체)` · `Project/Business(사업)` · 메모↔업체/사업 연결, 화면: 업체 목록·상세
  - **현재 데이터:** 사업명↔주관업체 연결만 존재. **담당자·연락처 데이터는 없음** → 2차 착수 시 담당자/연락처 입력·수집 체계가 선행 필요 (가장 큰 의존성)
- **1단계 — AI 챗봇**: 자연어로 묻고 답하는 업무 지원 ("XX 프로젝트 진행상황은?"), 반복 질문 대응으로 관리자 리소스 절약.
- **2단계 — 데이터 자동 정리**: 자연어 작성 시 AI 자동 분류, 축적된 이슈·대화 자동 요약, 프로젝트별 리포트 자동 생성, 패턴 분석 인사이트.

> 확장 대비: 메모 모델에 `project`·`tags`를 일관되게 유지하면 추후 고객/프로젝트 단위 집계·AI 요약에 그대로 활용 가능.
