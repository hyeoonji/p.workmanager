# WISM 기술 스택 및 시스템 아키텍처

## 0. 개발 단계 (Phase) 개요

| 단계 | 데이터 소스 | DB | 통신 | 비고 |
|------|-----------|-----|------|------|
| **Phase 1 (현재)** | Room (LocalDataSource) | SQLite (앱 내장) + Seed 테스트 데이터 | 없음 (로컬) | 앱 UI/기능 개발에 집중 |
| **Phase 2 (서버 연동)** | Retrofit (RemoteDataSource) | MySQL (사내 서버) | HTTPS REST API | Repository 구현체만 교체 |

> Phase 1 → 2 전환 시 **Repository 인터페이스는 동일**, DataSource 구현체(Hilt 모듈)만 교체하면 ViewModel/UI 코드 변경 없음.

## 2. 개발 단계별 로드맵

| 단계 | 환경 | 데이터 소스 | DB | 용도 |
|------|------|-----------|-----|------|
| **Phase 1: 로컬 개발** | 개발 PC / 에뮬레이터 | Room (LocalDataSource) | SQLite + Seed 데이터 | 앱 UI 개발 + 기능 구현 |
| **Phase 2: 서버 연동** | 사내 서버 (wims.win-tek.co.kr) | Retrofit (RemoteDataSource) | MySQL (사내 서버) | API 연동 + 운영 배포 |

### Phase 1 → Phase 2 전환 방법

```
변경 사항:
1. Hilt 모듈에서 DataSource 바인딩 교체
   - DatabaseModule (Room DAO 제공) → NetworkModule (Retrofit ApiService 제공)
   - RepositoryModule에서 LocalDataSource → RemoteDataSource로 바인딩 변경

2. 의존성 변경
   - Room 관련 의존성 제거 (또는 캐시용 유지)
   - Retrofit + OkHttp 의존성 추가

3. 설정 변경
   - API Base URL 설정 추가
   - JWT 토큰 인터셉터 추가 (OkHttp Interceptor)

변경 없음:
- Repository 인터페이스
- ViewModel 코드
- UI (Composable) 코드
- Navigation 코드
```

## 4. 아키텍처 패턴: MVVM + Repository

```
┌─────────────────────────────────────────────────────┐
│  View (Composable)                                  │
│  - ui/screens/, ui/components/                      │
│  - 사용자 인터랙션, UI 렌더링                          │
│       │ collectAsState (StateFlow 구독)              │
│       ▼                                             │
│  ViewModel (@HiltViewModel)                         │
│  - viewmodel/                                       │
│  - 비즈니스 로직, UI 상태 관리                         │
│       │ Repository 호출                              │
│       ▼                                             │
│  Repository (Interface)                             │
│  - data/repository/                                 │
│  - 데이터 소스 추상화                                 │
│       │                                             │
│       ├── LocalDataSource (Phase 1)                 │
│       │   - data/local/ (Room DAO)                  │
│       │                                             │
│       └── RemoteDataSource (Phase 2)                │
│           - data/remote/ (Retrofit ApiService)      │
└─────────────────────────────────────────────────────┘
```

| 패키지 | 역할 | 하는 일 |
|--------|------|--------|
| `ui/screens/` | **View** | Composable 화면 |
| `ui/components/` | **View** | 재사용 Composable 컴포넌트 |
| `ui/theme/` | **View** | 디자인 토큰 (Color, Type, Theme) |
| `viewmodel/` | **ViewModel** | 상태 관리, 비즈니스 로직 (@HiltViewModel) |
| `data/repository/` | **Repository** | 데이터 소스 추상화 (인터페이스 + 구현) |
| `data/local/` | **DataSource** | Room DAO, Entity (Phase 1) |
| `data/remote/` | **DataSource** | Retrofit ApiService, DTO (Phase 2) |
| `data/model/` | **Model** | 데이터 클래스 (Post, User, Comment 등) |
| `di/` | **DI** | Hilt 모듈 (DataSource 바인딩) |
| `navigation/` | **Navigation** | Navigation Compose 그래프 |

## 6. 인증 흐름

### Phase 1 (로컬)

```
[앱 시작] → DataStore에서 로그인 상태 확인
    │
    ├── 자동 로그인 ON + 저장된 userId → [P-02 홈 대시보드]
    │
    └── 로그인 정보 없음
                    ▼
             [P-01 로그인]
                    │ Room DB에서 아이디/비밀번호 확인
                    │ → DataStore에 userId + 자동로그인 플래그 저장
                    ▼
             [P-02 홈 대시보드]
```

### Phase 2 (서버)

```
[앱 시작] → DataStore에서 JWT 토큰 확인
    │
    ├── 토큰 없음 → [P-01 로그인] → POST /api/auth/login
    │                              → JWT 토큰 발급 + DataStore 저장
    │                              → [P-02 홈 대시보드]
    │
    └── 토큰 있음 → 토큰 만료 체크
         ├── 유효 → [P-02 홈 대시보드]
         └── 만료 → [P-01 로그인]
```

## 7. 기술 선택 이유

### Android Native (Kotlin)를 선택한 이유
- 사내 앱으로 **Android 전용** → 크로스 플랫폼 불필요
- Android API 직접 접근 가능 (알림, 백그라운드 등)
- Jetpack Compose로 선언적 UI 개발 가능
- Flutter 에뮬레이터/빌드 이슈 해소

### Jetpack Compose를 선택한 이유
- Google 공식 권장 모던 UI 프레임워크
- ViewModel/Navigation/Hilt와 자연스러운 통합
- XML 레이아웃 대비 보일러플레이트 대폭 감소
- Preview 기능으로 빠른 UI 확인

### Room DB (Phase 1)를 선택한 이유
- 서버 없이 즉시 앱 개발 가능
- 컴파일 타임 SQL 검증으로 안정성 확보
- Coroutines/Flow와 자연스러운 통합
- MySQL 스키마와 1:1 매핑으로 Phase 2 전환 용이

### Repository 패턴을 선택한 이유
- 데이터 소스 추상화로 Phase 전환 시 ViewModel/UI 변경 없음
- Hilt 모듈에서 바인딩만 교체하면 로컬 ↔ 서버 전환 완료
- 테스트 시 Mock 주입 용이

### JWT 인증을 선택한 이유 (Phase 2)
- 세션 관리 없이 stateless API 구현 가능
- 모바일 앱에 적합 (쿠키 의존성 없음)
